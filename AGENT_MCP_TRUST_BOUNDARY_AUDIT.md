# Agent/MCP Trust Boundary Audit

**Pilot price:** `$299`

For maintainers of agent frameworks, MCP servers, browser agents, memory systems, and agent-callable tools that have public attention but still need buyer-safe trust evidence before production adoption.

## The Gap

Many agent projects can show stars, forks, demos, docs, or directory attention. Buyers still need clearer answers:

```text
Can this agent touch secrets?
Can it spend money?
Can it message users?
Can it browse live sites?
Can it persist memory safely?
What evidence proves it did the right thing?
What must be approved before use?
```

## Good Fit

- Agent frameworks with tools, memory, browser use, schedules, delegates, or multi-agent execution.
- MCP servers that expose filesystem, browser, API, data, or workflow capabilities.
- Browser/computer-use tools where external side effects are possible.
- x402/agent-commerce endpoints where price, preflight, receipts, or paid-call boundaries are unclear.
- Open-source projects with public interest but no clear enterprise or buyer trust path.

## Deliverable

One public-data audit packet:

- trust-boundary map
- external-action inventory
- secrets and permission boundary review
- buyer-safe approval checklist
- proof/receipt gaps
- top 3 trust blockers
- one shippable README/docs patch outline
- one 7-day signal test

## What To Submit

Open the issue form and include only public information:

- repo or endpoint URL
- current signal such as stars, forks, directory rows, crawler hits, demo usage, search impressions, or issue interest
- current conversion boundary such as `paid_customers=0`, `pilot_requests=0`, `enterprise_inquiries=0`, `paid_calls=0`, or `production_adoptions=unknown`
- the riskiest capabilities: secrets, browser, memory, filesystem, payments, messaging, account changes, or code execution
- desired outcome: paid pilot, production-readiness proof, safer docs, accepted listing, buyer confidence, or partner review

Do not include API keys, passwords, private customer data, payment details, confidential logs, or private admin URLs.

## Boundaries

No guarantee of revenue, security certification, listing acceptance, or production safety.

This audit does not run exploit tests, scan third-party systems, bypass CAPTCHA/login/bot defenses, send outreach, create accounts, change payment providers, or make legal, financial, or compliance claims.

It produces a public-data trust-boundary plan and buyer-facing proof path.

## Fast Fit Check

If you are not sure whether the paid pilot fits, open the free public fit-check issue first. Good fit is usually visible when:

- public attention exists,
- the project has real agent/tool capabilities,
- a buyer would need trust evidence before production use,
- and the next step is a concrete proof artifact rather than more architecture.

## Why This Exists

Agent projects are increasingly powerful, but buyers do not only buy capability. They buy confidence:

```text
what can it do,
what can it not do,
who approves irreversible actions,
what does it cost,
what proof exists,
and how does it fail safely?
```

This audit turns that into a buyer-readable package.
